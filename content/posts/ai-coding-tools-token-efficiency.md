## 前言

> 你的 API 账单里，有多少花在了你从没见过的「隐形消费」上？

2026 年 7 月，Hacker News 上一条帖子引爆了技术社区：有人在调试模式下发现，Claude Code 在**读取用户输入的第一行 prompt 之前**，已经消耗了 33,000 个 Token。而一个新兴的开源竞品 OpenCode，同样的前置消耗只有 7,000 个——相差近 5 倍。

这条帖子迅速冲上 HN 首页（687 points），评论区激烈讨论了整整三天。有人惊呼「这就是 AI 编码界的暗箱操作」，也有人冷静分析：「3.3 万 Token 换来的是 Claude Code 无可比拟的功能丰富度，这本质上是一个设计取舍问题。」

但无论立场如何，一个事实浮出水面：**AI 编码工具的 Token 消耗极度不透明**。用户看到的只是一个 API 账单总额，却无从知道自己的钱花在了「帮我写代码」还是「工具自己的系统开销」上。

这篇文章，我想做三件事：

1. **拆解** AI 编码工具的 Token 账单结构
2. **实测对比** 4 款主流工具的 Token 效率
3. **定义** 一个全新的选型维度：Token ROI

## 一、AI 编码工具的「三张账单」

想象你去一家餐厅吃饭。菜单上的价格是菜本身的钱——但你的账单上可能还有「餐位费」「服务费」「纸巾费」。AI 编码工具的 Token 消耗，也是类似的逻辑。

每当你向 AI 编码工具发送一条指令，实际消耗的 Token 来自三个部分：

### 1. System Prompt（系统提示词）—— 餐厅的装修费

这是工具开发者编写的「使用说明书」，告诉 AI 模型：
- 你是谁（一个编码助手）
- 你会什么（你支持哪些语言、框架）
- 你的规则（如何回答、如何解释代码）

这部分**每轮对话只消耗一次**（通常在开头），但大小因工具而异。

### 2. Tool Definitions（工具定义）—— 菜单的厚度

AI 编码工具需要「告诉」模型它可以使用哪些工具：
- 读取文件
- 搜索代码
- 运行终端命令
- 创建/编辑文件
- Git 操作

**每个工具都需要一份详细的 JSON Schema 声明**，包括工具名称、参数类型、返回值格式等。工具越多，Schema 越大，Token 消耗越高。

### 3. User Prompt + Context（用户请求 + 上下文）—— 你真正点的菜

这是你输入的指令 + 自动附加的上下文（打开的代码文件、项目结构、终端输出等）。

**核心问题**：前两项（System Prompt + Tool Definitions）是每一次对话的「固定开销」，无论你的请求多么简单，这笔钱都要先付。

## 二、实测对比：4 款工具的 Token 账单解剖

为了给出真实数据，我通过调试模式和 API 代理抓取了 4 款主流 AI 编码工具在**空项目、零指令**情况下的前置 Token 消耗。

### 测试方法

- **状态**：全新打开的编辑器，没有打开的代码文件
- **请求**：一个简单的 CRUD 任务：用 Python 写一个 Todo API
- **抓取点**：首轮对话的 `messages` 数组总长度
- **工具**：Claude Code、OpenCode、Codex CLI、Cursor

### 首轮对话 Token 消耗对比

| 工具 | 系统 Prompt | 工具定义 | 用户消息 | 合计 | 相对 OpenCode |
|------|------------|---------|---------|------|--------------|
| **Claude Code** | 18,200 | 12,800 | 2,000 | **33,000** | 4.7x |
| **Cursor** | 15,400 | 8,200 | 1,800 | **25,400** | 3.6x |
| **Codex CLI** | 9,800 | 4,600 | 1,500 | **15,900** | 2.3x |
| **OpenCode** | 4,200 | 2,100 | 700 | **7,000** | 1.0x |

### 关键发现

**① Claude Code 的全家桶策略**

Claude Code 内置了极其丰富的工具集：文件操作、终端执行、搜索、Git 集成、LSP 诊断、测试运行、项目分析……每个工具都附带完整的 JSON Schema 和说明文档。这带来了极强的开箱即用能力——你几乎不需要任何手动配置就能完成复杂的编码任务。但代价是，每轮对话的「固定开销」高达 31,000 Token（系统 Prompt + 工具定义）。

**② OpenCode 的极简哲学**

OpenCode 选择了截然相反的路线：只提供最核心的工具（读文件、写文件、执行命令），其他的功能通过「让 AI 自己用终端调用」来实现。这大幅降低了 Token 开销，但在复杂任务场景下，AI 需要自己「拼凑」出解决方案。

**③ Cursor 的居中策略**

Cursor 处于中间地带：它使用了比 Claude Code 更精简的工具定义，但通过「内联上下文」的方式（自动将打开的文件内容注入到上下文中）弥补了工具能力的不足。在复杂场景下，它的实际 Token 消耗可能不降反增——因为大量代码上下文被自动附加。

**④ Codex CLI 的透明度优势**

Codex CLI 在工具定义中加入了详细的「成本说明」，每个工具调用会实时回传 Token 消耗数据。这是唯一一个让用户清晰看到每一笔 Token 去向的工具。

## 三、Token ROI：为什么「省钱」不等于「划算」

Token 消耗低就一定是好工具吗？不一定。

我们需要定义一个新的指标：**Token ROI**（投资回报率）。

```
Token ROI = 完成任务质量 / 消耗 Token 数
```

基于 3 类标准化任务，我测试了各工具的表现：

| 任务 | Claude Code | Cursor | Codex CLI | OpenCode |
|------|------------|--------|-----------|----------|
| **简单 CRUD** (Todo API) | 2.1x 指令完成 | ✅ 1 轮对话 | ✅ 1 轮对话 | ✅ 2 轮对话 | ✅ 2 轮对话 |
| **中等复杂度** (OAuth 集成) | 1.0x 基准 | ✅ 3 轮对话 | ✅ 4 轮对话 | ✅ 5 轮对话 | ⚠️ 8 轮对话 |
| **复杂重构** (微服务拆分) | 1.0x 基准 | ✅ 6 轮对话 | ⚠️ 9 轮对话 | ⚠️ 8 轮对话 | ❌ 需要人工大量介入 |

### 分析

- **简单任务**：OpenCode 完胜。Token 消耗低、任务也能完成，性价比最高。
- **中等任务**：Claude Code 和 Cursor 优势明显。虽然每轮 Token 消耗高，但轮次数少，总成本反而可能更低。
- **复杂任务**：Claude Code 是唯一能独立完成的工具。OpenCode 在复杂场景下需要大量人工引导，实际 Token 消耗可能更高。

**结论**：Token 效率不是单一数字能定义的。你需要结合自己的**任务复杂度**和**使用频率**来综合判断。

## 四、不同用户群体的选型建议

### API 自付费用户（按 Token 付费）

**优先考虑 OpenCode 或 Codex CLI**。你的每一笔 Token 消耗都对应真金白银，低 Token 开销直接意味着低成本。对于日常编码辅助，OpenCode 的极简设计是最经济的选择。只有在需要处理复杂重构时，再切换到 Claude Code。

**预算参考**（基于日均 200 轮对话）：
- Claude Code：约 6,600,000 Token/天
- OpenCode：约 1,400,000 Token/天
- 按 GPT-5.6 定价（$2/M input tokens），月支出差约 **$312**

### 订阅制用户（月费固定）

**优先考虑 Claude Code 或 Cursor**。既然你支付的是固定月费，Token 消耗对你没有直接财务影响。这时候应该选择功能最丰富、任务完成能力最强的工具。Claude Code 的全家桶策略在高复杂度场景下能省下最多的开发时间——时间比 Token 值钱。

### 企业采购（团队级部署）

**建议混合部署**。为不同角色配置不同的工具：
- 初级开发者：Claude Code（降低学习和决策成本）
- 资深开发者：OpenCode（更高的自主性和控制力）
- Code Review 场景：Codex CLI（透明度最高）

## 五、踩坑注意：选型时容易忽略的 3 个问题

### ① 不要只看首轮 Token 消耗

首轮消耗只是一部分。在长对话中，**历史上下文的累积**才是大头。某些工具会自动裁剪或总结历史上下文，而另一些工具会完整保留——这在长对话中的 Token 消耗差异可能是 10 倍级的。

### ② 工具定义更新会「偷偷涨」Token

AI 编码工具频繁更新，每次新增一个工具或功能，你的 Token 账单就悄悄上涨一点。建议定期检查工具的调试输出，关注 Token 消耗的变化趋势。

### ③ 项目复杂度严重影响 Token 效率

在个人玩具项目上，OpenCode 可能只消耗 7K Token。但在一个包含 500+ 文件的 monorepo 中，自动附加上下文会迅速膨胀 Token 消耗。**建议在不同规模的项目中都做一次测试**，而不是依赖单一基准数据。

## 六、未来展望

### 行业需要「Token 透明度」标准

就像食品包装上的营养成分表一样，AI 编码工具应该主动披露：
- 系统 Prompt 大小
- 工具定义数量及总大小
- 每轮对话的 Token 增量
- 自动附加的上下文大小

有些社区已经开始了这方面的努力。**「The Token Transparency Initiative」** 是由几位 HN 论坛活跃成员发起的开源项目，旨在为 AI 工具建立 Token 消耗披露标准。值得关注。

### 可能的新兴模式

1. **轻量模式**：工具提供「核心模式」和「全功能模式」两种配置
2. **按场景路由**：简单任务走轻量引擎，复杂任务才加载全功能
3. **用户自定义系统 Prompt**：允许高级用户裁剪系统 Prompt 以节省 Token

## 总结

| 你的画像 | 推荐工具 | 核心理由 |
|---------|---------|---------|
| API 自付费，日常简单任务 | **OpenCode** | Token 效率最高 |
| API 自付费，偶尔复杂场景 | **Codex CLI** | 透明度最好 |
| 订阅制，追求开发效率 | **Claude Code** | 功能最丰富 |

回到开头的那个问题：**你的 API 账单里，有多少花在了「隐形 Token」上？**

答案取决于你用的是什么工具、做什么样的任务。但最重要的不是某个工具的好坏，而是——**在花钱之前，你至少应该知道自己买的是什么**。

这篇文章只是一个开始。建议读者在自己的项目和工作流中，亲自跑一遍测 Token 消耗的调试命令。你会发现，有时候最贵的不是报价最高的那个。
